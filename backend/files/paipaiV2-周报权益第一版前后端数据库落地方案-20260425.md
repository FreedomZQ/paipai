# paipaiV2 周报权益第一版前后端数据库落地方案（2026-04-25）

> 目标：在现有拍拍伴读前后端基础上，完成第一版周报权益升级：免费用户不展示高级预览；标准用户只支持单孩子家庭，也不展示家庭高级预览；标准用户和高级用户支持历史报告查看；不支持导出；支持页面分享；历史报告生成内容需要在 App 本地留存一份，确保后端服务掉线时用户仍可查看已生成历史报告预览；所有模块可见性和孩子数量权益以后端/数据库配置为准。

---

## 0. 当前结论

当前项目已经有周报基础能力：

- 后端：`GET /api/v1/reports/weekly/current`、`GET /api/v1/reports/weekly/history`
- 前端：`WeeklyReportView.swift`、`WeeklyParentReport.swift`、`BackendClient.fetchCurrentWeeklyReport(...)`
- 权益：`reading_plan_catalog` 中已有 `free` 和 `family_multi_child_lifetime`
- 孩子数量限制：现有逻辑已经通过后端 `AccountEntitlementView.childLimit` 控制，来源于数据库 `reading_plan_catalog.childLimit`

但当前不足：

1. 只有免费版 + 一个家庭版，缺少明确的“标准单孩子付费版”。
2. 现有 `family_multi_child_lifetime` 把 `family_access` 和 `family_multi_child` 都作为别名，导致月付/年付这类 `family_access` 会被归入家庭多孩子权益。
3. 周报返回结构不是模块化，无法精确控制 free / standard / family 各自能看到什么。
4. 当前计划里曾考虑 locked preview，但本次要求：free 不展示高级预览；standard 不展示家庭预览。
5. 第一版支持标准/高级用户查看历史周报，不做 PDF/图片/文件导出。
6. 历史周报需要支持 App 本地留存：后端服务不可用时，用户仍能看到已成功生成并缓存过的历史报告预览。

## 1. 第一版权益规则

### 1.1 用户类型

第一版三类用户：

| 用户类型 | planCode 建议 | entitlementCode 建议 | 孩子数 | 周报范围 | 历史 | 导出 | 页面分享 |
|---|---|---|---:|---|---|---|---|
| 免费 | `free` | `free` | 1 | 单孩子基础 | 不支持 | 不支持 | 支持 |
| 标准付费 | `standard_single_child` | `family_access` 或 `standard_access` | 1 | 单孩子完整 | 支持最近 4 周 | 不支持 | 支持 |
| 家庭高级 | `family_multi_child_lifetime` / `family_pro` | `family_multi_child` | DB 配置，默认 5 | 家庭 + 单孩子 | 支持最近 12 周 | 不支持 | 支持 |

> 注意：如果当前 App Store 商品已经存在 `com.paipai.readalong.family.monthly` / `yearly` 并映射到 `family_access`，可以先把 `family_access` 定义为“标准单孩子付费权益”。后续如要改名为 `standard_access`，需要同步 App Store、后端 mapping、数据库历史交易权益投影。

### 1.2 模块可见性

第一版不做预览：

| 模块 | free | standard | family |
|---|---|---|---|
| `basic_stats` 基础统计 | full | full | full |
| `safe_summary` 安全总结 | full | full | full |
| `basic_suggestions` 基础建议 | full，最多 2 条 | full | full |
| `single_child_detail` 单孩子详细回顾 | hidden | full | full |

| `offline_history_preview` 离线历史预览 | hidden | full，本地缓存已生成报告 | full，本地缓存已生成报告 |
| `export_report` 导出 | hidden | hidden | hidden |
| `page_share` 页面分享 | full | full | full |

关键要求：

- free 不展示“升级后可看什么”的模块预览，只保留底部轻 CTA 即可。
- standard 不展示家庭模块预览。
- 历史周报第一版仅对 standard / family 开放；free 不开放历史，也不展示历史预览。
- 历史周报需要在 App 本地保存已生成内容，用于后端服务掉线时的历史预览。
- 导出能力第一版全档关闭。
- 页面分享不是导出：只分享当前页面摘要文本或 App 链接，不生成 PDF/图片/文件。

### 2.3 注意：SysRemoteConfigService 当前无唯一约束

当前 `sys_remote_config` 读取逻辑是：

- 按 `app_code + namespace_code + status = active` 查所有配置。
- `updated_at DESC, id DESC` 排序。
- 同 key 用第一条。

因此 migration 可以直接追加新 active 行，新行会覆盖旧行。也可以选择把同 key 老行置为 inactive，再插新行；生产更推荐先 inactive 再 insert，避免同 key active 过多。

### 3.4 改造 `weeklyHistory(...)`

第一版支持 standard / family 查看历史周报，free 不支持历史。

推荐逻辑：

```java
public WeeklyReportHistoryView weeklyHistory(...) {
    AccountEntitlementView entitlement = accountState(user).entitlement();
    WeeklyReportAccessConfig access = weeklyReportAccessConfigService.current();
    int historyWeeks = access.historyWeeksFor(entitlement.planCode());
    if (historyWeeks <= 0) {
        return new WeeklyReportHistoryView(scope == null ? "child" : scope, childId, false, 0, 0, List.of());
    }
    return buildWeeklyHistory(user, childId, scope, historyWeeks);
}
```

历史生成方式：

- standard：只生成单孩子最近 4 周。
- family：单孩子最近 12 周；`scope=family` 时生成家庭最近 12 周。
- free：返回空列表，不报错，避免老客户端崩。

第一版可以先实时聚合最近 N 周，不必新建 snapshot 表。后续如访问量上升，再增加 `reading_weekly_report_snapshot`。

### 3.5 确认标准用户只支持 1 个孩子

现有逻辑已经支持通过 DB 改孩子数量：

- 创建孩子：`ReadingCompatService.createChild(...)`
  - 用 `accountState(user).entitlement().childLimit()` 拦截。
- 已下线多端数据方案：`Reading已下线多端数据方案Validator`
  - 用 `state.entitlement().childLimit()` 拦截。
- 前端孩子管理：`ManageChildrenView.swift`
  - 用 `appState.accountState?.entitlement.childLimit` 展示和阻止。

需要确保：

1. `standard_single_child.childLimit = 1`
2. `standard_single_child.multiChildEnabled = false`
3. `standard_single_child.weeklyReportScope = child`
4. `standard_single_child.weeklyReportHistoryWeeks = 4` 且 `historyEnabled = true`
5. `family_multi_child_lifetime.weeklyReportHistoryWeeks = 12` 且 `historyEnabled = true`
6. `family_multi_child_lifetime.matchedEntitlementCodes` 不再包含 `family_access`

否则 `family_access` 仍可能被识别为家庭多孩子。

### 3.6 现有排序逻辑风险

`resolvePlanCatalogEntries()` 当前会按：

1. premium 优先
2. childLimit 大优先
3. localCardLimit 大优先
4. planCode 字典序

如果 `family_multi_child_lifetime` 仍有 `matchedEntitlementCodes = ["family_access", "family_multi_child"]`，那么 `family_access` 会优先命中家庭版，因为家庭版 childLimit 更大。

必须改成：

- standard：`matchedEntitlementCodes = ["family_access"]`
- family：`matchedEntitlementCodes = ["family_multi_child"]`

这样才能保证标准用户只支持 1 个孩子。

```swift
offlineHistoryPreviewEnabled = try container.decodeIfPresent(Bool.self, forKey: .offlineHistoryPreviewEnabled) ?? historyEnabled
```

后端 `AccountEntitlementView` 也需要加对应字段，或者第一版只从周报接口字段读取。

### 4.3 AppState 请求历史并落本地缓存

文件：

```text
ios/PaipaiReadAlongV2/App/PaipaiReadAlongV2App.swift
```

第一版 standard / family 支持历史周报。历史报告采用 **cache-first**：先看本地是否已有对应账号 + plan + scope + child 的报告缓存；有缓存直接展示，避免每次打开都请求后端；没有缓存再调用 `/history`，成功后保存本地。

推荐流程：

```swift
if accountState?.entitlement.historyEnabled == true,
   let accountId = accountState?.accountId {
    let historyScope = accountState?.entitlement.weeklyReportScope == "family" ? "family" : "child"
    let historyChildId = historyScope == "family" ? nil : resolvedChildId
    history = weeklyReportCache.load(
        accountId: accountId,
        planCode: accountState?.entitlement.planCode,
        childId: historyChildId,
        scope: historyScope
    )
    historyLoadedFromCache = history != nil
    if history == nil {
        history = try await backendClient.fetchWeeklyReportHistory(childId: historyChildId, scope: historyScope)
        if let history {
            weeklyReportCache.save(history, accountId: accountId, planCode: accountState?.entitlement.planCode, childId: historyChildId, scope: historyScope)
        }
    }
}
```

家庭周报仍可按权益请求：

```swift
if accountState?.entitlement.weeklyReportScope == "family" {
    familyReport = try? await backendClient.fetchCurrentWeeklyReport(childId: nil, scope: "family")
}
```

### 4.4 本地历史报告缓存设计

新增文件建议：

```text
ios/PaipaiReadAlongV2/Core/Services/WeeklyReportLocalCache.swift
```

第一版低运维实现：

- 使用 App sandbox 内 JSON 文件缓存，不进 iCloud，不导出。
- 每个用户、plan、scope、childId 独立 key。
- 只缓存后端已经返回的周报内容，不在本地伪造权益。
- free 用户不读取历史缓存，避免降级/越权后继续看付费历史。
- standard 缓存最近 4 周。
- family 缓存最近 12 周。
- 后端掉线时展示“离线预览”提示。

缓存 key：

```text
weekly-report-v1/{accountId}/{planCode}/{scope}/{childId-or-family}.json
```

缓存内容：

```json
{
  "cachedAt": "2026-04-25T04:17:00Z",
  "planCode": "standard_single_child",
  "scope": "child",
  "history": {}
}
```

安全边界：

- 本地缓存只用于“已生成历史报告预览”。
- 当前 accountState 不具备 history 权益时，不展示缓存。
- 用户退出登录、删除账号、切换账号时清理对应缓存。
- 不缓存原始图片、音频、OCR 全文，只缓存周报摘要和统计。

## 5. 后端掉线时的历史预览策略

### 5.1 服务端职责

后端负责生成权威周报内容，返回：

- 当前周报：`/api/v1/reports/weekly/current`
- 历史周报：`/api/v1/reports/weekly/history`
- 权益字段：`historyEnabled`、`weeklyReportHistoryWeeks`、`offlineHistoryPreviewEnabled`

后端不依赖客户端缓存判断权益。客户端缓存只是可用性兜底。

### 5.2 App 本地留存职责

App 在 standard / family 用户成功获取历史周报后，将完整响应保存到本地 JSON 文件。

当后端服务掉线、超时、DNS 失败、网关 5xx 时：

1. 检查当前 `accountState.entitlement.historyEnabled == true`。
2. 检查本地是否有同 accountId + planCode + childId/scope 的历史缓存。
3. 有缓存则展示历史预览，并显示离线提示。
4. 无缓存则展示“历史报告暂时无法加载，联网后会自动刷新”。

### 5.3 权益变更和缓存处理

- free 用户不展示历史缓存。
- standard 用户只展示 child scope 缓存，最多 4 周。
- family 用户可展示 child / family scope 缓存，最多 12 周。
- 用户退出登录、删除账号、切换账号时清理缓存。
- 如果用户从 family 降级到 standard，不展示 family scope 缓存。
- 如果用户从 standard 降级到 free，不展示任何历史缓存。

### 5.4 隐私和合规

本地缓存不保存原始图片、音频、OCR 全文、儿童敏感原始材料，只保存：

- 周期
- 聚合统计
- 安全总结
- 建议文案
- disclaimer

本地缓存仍属于用户设备内数据，删除账号/退出登录时应清除。

## 7. 验收清单

### 7.1 数据库验收

```sql
SELECT config_key, config_value_json

WHERE app_code = 'paipai_readingcompanion'
  AND namespace_code = 'reading_plan_catalog'
  AND status = 'active'
```

必须看到：

- `standard_single_child.childLimit = 1`
- `standard_single_child.multiChildEnabled = false`
- `standard_single_child.weeklyReportScope = child`
- `standard_single_child.historyEnabled = true`
- `standard_single_child.weeklyReportHistoryWeeks = 4`
- `standard_single_child.offlineHistoryPreviewEnabled = true`
- `standard_single_child.exportReportEnabled = false`
- `family_multi_child_lifetime.matchedEntitlementCodes` 不包含 `family_access`
- `family_multi_child_lifetime.childLimit` 来自 DB，默认 5，可改
- `family_multi_child_lifetime.historyEnabled = true`
- `family_multi_child_lifetime.weeklyReportHistoryWeeks = 12`
- `family_multi_child_lifetime.offlineHistoryPreviewEnabled = true`

### 7.2 后端验收

1. free 用户：
   - `/api/v1/reports/weekly/current?scope=child` 只返回基础模块。
   - `scope=family` 返回 403 或不允许。
   - `/history` 返回空。

2. standard 用户：
   - 权益返回 `childLimit = 1`。
   - 创建第二个孩子失败，错误为 `CHILD_LIMIT_REACHED`。
   - 周报只返回单孩子模块。
   - 返回 `history_reports`，最多最近 4 周。
   - 不返回 `family_overview`、`child_summaries`、`export_report`。
   - 后端异常时，App 可读取本地已缓存的历史报告预览。

3. family 用户：
   - 权益返回 DB 配置的 `childLimit`，默认 5。
   - `scope=family` 可以返回家庭模块。
   - `/history` 返回最近 12 周历史。
   - 不返回导出模块。
   - 后端异常时，App 可读取本地已缓存的历史报告预览。

4. 修改 DB 中 `family_multi_child_lifetime.childLimit` 为 3 后：
   - 用户重新请求 `/account/state` 或触发权益刷新后，前端显示孩子上限 3。
   - 后端创建孩子按 3 拦截。

### 7.3 前端验收

1. free：
   - 周报页无高级预览卡。
   - 无历史入口。
   - 不读取、不展示历史缓存。
   - 无导出按钮。
   - 有页面分享按钮。

2. standard：
   - 周报页无家庭预览卡。

3. family：

## 8. 是否可以随时调整不同用户支持孩子个数

结论：现有主链路已经可以通过数据库配置调整。

依据：

1. `ReadingCompatService.resolveEntitlement(...)` 根据用户 active entitlementCode 匹配 `reading_plan_catalog`。
2. `entitlementFromConfig(...)` 从配置读取 `childLimit`。
3. `createChild(...)` 使用 `state.entitlement().childLimit()` 做后端权威拦截。
4. `Reading已下线多端数据方案Validator` 也用同一个 `childLimit` 校验同步写入。
5. 前端 `ManageChildrenView.swift` 用 `accountState.entitlement.childLimit` 展示和拦截。

因此：

- 改 `standard_single_child.childLimit` 可以调整标准版孩子数。
- 改 `family_multi_child_lifetime.childLimit` 可以调整家庭版孩子数。
- 不需要发版即可改后端权益；但前端需要重新拉取 account state 才能显示最新值。

需要注意：

- 如果对 `reading_plan_catalog` 也引入缓存，需要提供同类 refresh 接口。
- 当前 `SysRemoteConfigService` 本身没有缓存，每次 loadNamespace 会读 DB，所以孩子数量调整通常下一次请求即可生效。
- 如果用户已经创建了超过新上限的孩子，后端应只阻止新增，不建议自动删除已有孩子。前端应提示“当前权益最多新增/保留 X 个活跃档案，如需继续新增请升级”。
