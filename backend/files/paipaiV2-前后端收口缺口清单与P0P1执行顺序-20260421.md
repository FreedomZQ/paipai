# PaipaiV2 前后端收口缺口清单与 P0/P1 执行顺序

> ⚠️ 历史说明（2026-04-22）：本文件记录的是 2026-04-21 对“尚未收口状态”的排查结论。当前 reading 已继续推进并收口为 **Apple 登录唯一正式入口**；删除账号仍保留**临时输入邮箱验证码确认**，但 demo session / 邮箱登录已不再是现行目标方案。

更新时间：2026-04-21 23:40（Asia/Shanghai）

## 1. 结论摘要

**结论：当前 paipaiV2 前端与 unified backend 已经具备“主链路大体打通、可继续联调”的基础，但还没有达到“完全匹配、无冗余、可直接上线”的标准。**

### 当前判断
- **接口主链路匹配度：高**
- **已下线多端数据方案 多 App 通用后端骨架：已具备**
- **新需求覆盖度：部分已落地，部分仅落到一半**
- **上线状态：未达到上线标准**

### 为什么还不能判定为上线标准
核心原因不是“完全没做”，而是**还有 6 个会直接影响产品边界、合规或上线闭环的 P0 缺口**：
1. 登录/跳过/guest 与 已下线多端数据方案 无游客方案互相冲突
2. 设备信息在用户同意前就上报
3. Translation 能力检查没有真正闭环，且 macOS 版本要求与需求不一致
4. 公告弹窗能力后端字段有了，但前端没有把规则与动作真正吃透
5. 使用时长 30 天留存策略没有真正前后端闭环
6. Release 配置与真实构建闭环仍未收口

## 4. P0 缺口清单（上线前必须收口）

## P0-1 登录/跳过/guest 路径与 已下线多端数据方案“无游客账户”方案冲突

### 现状
已下线多端数据方案 方案文档已经明确：
- 第一版 **没有游客账户**
- 所有同步数据绑定正式登录用户

但当前代码仍保留了完整 guest/demo 路径：
- 前端 onboarding 支持“跳过”
  - `Features/Onboarding/OnboardingView.swift:41`
- （历史问题）前端启动时曾自动创建 demo session，现目标是彻底移除该路径
  - `App/PaipaiReadAlongV2App.swift:225-226`
- 前端 `AppBootstrap` 仍有 `guestModeEnabled`
  - `Core/Models/AppBootstrap.swift:5,18`
- 后端 app-definition 仍开启 `demoSessionEnabled: "true"`
  - `src/main/resources/apps/reading/app-definition.yml:26-27`
- 后端 bootstrap 直接把 `guestModeEnabled` 返回为 `true`
  - `ReadingCompatService.java:143-145`

### 影响
这是**架构层冲突**，会直接影响：
- 数据归属
- 是否允许已下线多端数据能力
- 本地数据是否需要 guest -> formal 合并
- 公告、使用时长、设备事件、付费等行为边界

### 判断
**这是 P0。必须先选一个产品边界，不然前后端永远收不干净。**

### 待办
- 二选一：
  1. **坚持无 guest**：去掉 skip 后可进入学习主链路的能力，改为必须登录后才能进入同步态
  2. **保留 skip/local guest**：明确 guest 仅本地可用、仅限本机、不可恢复、不可多端合并，并补完整 guest -> formal 合并策略
- 若按当前 已下线多端数据方案 文档执行，建议：
  - 关闭 `demoSessionEnabled`
  - `bootstrap.guestModeEnabled=false`
  - 前端去掉启动自动创建 demo session（当前口径已定）
  - Onboarding 的“跳过”改成“仅浏览，不进入学习数据主链路”或直接改为“稍后登录但先浏览说明页”

## P0-2 设备信息在用户明确同意前就已上报，合规边界未闭环

### 现状
需求要求：
- 首次启动或首次登录时通过弹窗提示用户阅读并明确同意隐私政策
- 设备信息收集需告知并同意

当前前端在启动时会直接上报 `app_launch`：
- `App/PaipaiReadAlongV2App.swift:207-208`

同意隐私后也会单独上报 `privacy_accepted`：
- `App/PaipaiReadAlongV2App.swift:482-483`

也就是说：
- **同意前已经上报了 device info**
- `DeviceInfoService.requestPrivacyConsent()` 存在，但没有形成真正拦截链路

### 影响
这是明显的**合规闭环缺口**：
- 代码行为与需求不完全一致
- 也与前端 onboarding 里的“先说明再同意”叙述不一致

### 判断
**P0。上线前必须统一成“先同意，再上报非必要设备事件”。**

### 待办
- 将 `app_launch` 上报拆分为：
  - `app_open_basic`（不含设备细粒度字段，或仅本地）
  - `device_profile_collected`（用户同意后才上报）
- 在前端启动阶段加拦截：未同意隐私前，不发送设备型号/系统版本类事件
- 后端 `ReadingDeviceEventService` 增加事件白名单与 payload 校验，避免前端误传也落库

## P0-3 Translation 功能可用性检查没有真正落地，且 macOS 要求与新需求不一致

### 现状
需求明确要求：
- iOS 17.4+
- Mac 上大部分能力 macOS 13+ 可实现
- 如果系统“翻译”App 被删除，要正确报错
- 语言包是按需下载，需做可用性检查

当前代码问题：
1. `TranslationService` 把 macOS 可用版本写成了 **14.4+**
   - `Core/Services/TranslationService.swift:20,39`
2. `checkLanguagePackAvailability()` 只是简单返回 `true`
   - `Core/Services/TranslationService.swift:37-44`
3. `downloadLanguagePackIfNeeded()` 只是调用 availability，没有下载动作
   - `Core/Services/TranslationService.swift:46-47`
4. `fallbackTranslate()` 会把“未真正翻译”的结果伪装成成功
   - `Core/Services/TranslationService.swift:50-55`
5. 这些检查函数当前没有被真正调用
   - 全项目仅定义，无实际消费

### 影响
这会导致：
- Mac 13 兼容承诺与代码不一致
- 用户删掉系统翻译 App 或语言包缺失时，前端不能给出正确引导
- “翻译失败”会被伪装成“翻译成功”，非常危险，容易把测试结果误判成通过

### 判断
**P0。因为这是新增需求的核心能力闭环。**

### 待办
- 改正 macOS availability 判定，使其与产品目标一致
- 把 Translation 可用性检查做成真实能力判断，而不是返回 true
- 首次翻译前增加：
  - Translate 可用性检查
  - 语言包是否存在/是否需下载
  - 下载/失败提示 UI
- 删除 `fallbackTranslate()` 这种“假成功”路径，至少要返回失败态
- 在 LearningDetail 首次进入时，把可用性检查纳入状态机

- `targetPlanCode`
- `targetMinAppVersion`
- `targetMaxAppVersion`
- `actionUrl`
- `actionText`

  - `App/PaipaiReadAlongV2App.swift:30`

  - `Features/Components/AnnouncementView.swift:59-65`
- 公告接口要求已登录
  - `ReadingAnnouncementCompatController.java:44`

### 影响
当前只能说：
- **“公告弹窗基础能力已通”**
- 但还不能说：
- **“后台可灵活配置规则、前端完整执行规则、打开 App 即按规则弹出”**

### 判断
**P0。因为这是新增需求明确点名的能力。**

- `AnnouncementManager` 增加 actionUrl 打开逻辑
- 根据产品决策决定：未登录是否也允许拉取“公共公告”

## P0-5 使用时长“30 天留存自动删除”没有前后端闭环

### 现状
前端有一个本地 `UsageTrackingService`，带 30 天 retention：
- `Core/Services/DeviceInfoService.swift:137+`

但它目前是**未接入冗余代码**：
- `App/PaipaiReadAlongV2App.swift:74` 仅实例化，未真正参与主流程

当前真正使用的是 已下线多端数据方案 / usage_session_v2：
- 前端学习页、拍读页、复习页会 start/end usage session
- 后端 `ReadingUsageService` 负责聚合 daily summary

问题在于：
- 代码里**未发现服务端 30 天清理策略**
- 也未发现同步侧对过期 usage 数据的统一清理任务
- 现在更接近“能累计”，但不是“按 30 天合规留存自动删除”

### 影响
需求要求的是：
- 每次和每天累计时长
- 明确留存期（如 30 天）
- 到期自动删除

当前只满足了前两项的一部分，第三项没真正闭环。

### 判断
**P0。因为它属于新增需求里的数据留存要求。**

### 待办
- 明确 retention owner：后端权威还是本地权威
- 若以后端为准：
  - 增加定时清理 job，删除 30 天前的 `reading_usage_session_v2`
  - 同步修正 `reading_child_usage_daily` 的可追溯边界
- 若以客户端为准：
  - 同步上传前过滤过期 usage session
  - 后端上传侧增加过期拒绝/清理
- 删除或重构未使用的 `UsageTrackingService`，避免双轨统计

## 7. 建议执行顺序（按 P0 -> P1）

## 第一阶段：P0，先把产品边界和合规闭环定死

### Step 1. 先定 auth 边界（最高优先级）
1. 最终拍板：**是否允许 skip/guest**
2. 如果不允许：
   - 关闭 demo session（当前口径已定）
   - bootstrap 改为 guestModeEnabled=false
   - 去掉启动自动创建 demo session（当前口径已定）
3. 如果允许：
   - 补 guest 与 已下线多端数据方案 的隔离规则
   - 明确 guest 数据不进云 / 不可恢复 / 不做多端同步

### Step 2. 收口隐私同意与设备事件上报顺序
1. 前端未同意前不再上报细粒度设备信息
2. 后端为 device-event 增加字段级校验
3. 更新法务与测试用例

### Step 3. 收口 Translation 真可用性
1. 修正 macOS availability
2. 做翻译 App / 语言包检查
3. 去掉假成功 fallback
4. 在学习页给出真实错误提示和引导

### Step 4. 收口公告系统完整规则
1. 拉取接口补 scene / locale / appVersion / planCode
2. actionUrl 真正可点击执行
3. 决定未登录用户能否收公共公告

### Step 5. 收口使用时长 retention
1. 明确 retention owner
2. 增加 30 天清理策略
3. 删除冗余本地 usage 方案

### Step 6. 收口 release gate
1. 注入真实 release config
2. 补齐 Apple / App Store 生产配置
3. macOS/Xcode 真编译
