# 统一后台式权益中心升级方案 - 模式三老用户权益保护版

## 1. 本次确认的目标

本方案以“模式三”为最终目标：

```text
权益增加：老用户也自动获得新增权益。
权益减少：不能影响已付款老用户原本购买时拥有的权益，避免法律纠纷和用户投诉。
免费权益：可以按当前后台配置动态调整。
付费权益：必须保留购买时权益快照，并支持用户级补偿、赠送和覆盖。
前端 App：尽量不改或少改，主要通过后端权益接口返回兼容字段实现。
```

本次升级不建议只做“会员等级配置中心”，必须同时补齐三项用户级能力：

1. 用户级赠送/补偿权益。
2. 用户级功能覆盖。
3. 购买时权益快照。

最终后端判断用户权益时，不能只看当前会员等级配置，而要合并：

```text
当前会员等级配置
+ 用户购买时权益快照
+ 后台新增权益策略
+ 用户级赠送/补偿权益
+ 用户级功能覆盖
= 用户最终可用权益
```

## 2. 当前问题

当前系统已经有用户级付款记录和权益记录：

```text
sys_purchase_transaction
sys_entitlement_snapshot
```

但是当前 `sys_entitlement_snapshot` 只记录：

```text
app_code
user_id
entitlement_code
status
source_type
expires_at
payload_json
```

它没有结构化记录：

```text
用户购买时属于哪个 planCode
用户购买时具体拥有了哪些 feature
用户购买时每个 feature 的额度是多少
用户购买时使用的是哪个权益版本
后续权益减少时是否要保护老用户
```

因此当前逻辑存在风险：

```text
用户购买 standard_single_child 时有 4 周历史周报
后来后台把 standard_single_child 改成 2 周历史周报
老用户下次查权益也会变成 2 周
```

这会造成“已付款承诺权益被减少”的风险。

---

## 3. 改造后的权益合并原则

### 3.1 总原则

用户最终权益按以下优先级合并：

```text
第 1 层：当前 App 当前会员等级配置
第 2 层：购买时权益快照保护
第 3 层：权益增加自动补齐
第 4 层：用户级赠送/补偿权益
第 5 层：用户级功能覆盖
```

最终规则：

```text
布尔型权益：只在有效购买快照内，老快照或当前配置任一为 true，则保留 true，除非用户级覆盖明确关闭。
额度型权益：只在有效购买快照内，取购买快照额度和当前配置额度中的较大值，除非用户级覆盖明确指定。
周期型权益：只在有效购买快照内，取购买快照周期和当前配置周期中的较大值。
模块型权益：只在有效购买快照内，老快照中 full 的模块不能因为当前配置变 hidden 而被关闭；当前新增 full 模块可以给老用户补上。
购买快照过期后：历史快照不再参与合并；用户再次购买或续费成功时，必须按当时最新后台权益配置生成新的购买快照。
```

### 3.2 权益有效期原则

购买时权益快照不是永久保护，它只能保护该次购买或该个订阅周期内的权益。

```text
快照有效条件：
status = active
starts_at <= now
expires_at is null 或 expires_at > now
```

其中：

```text
订阅商品：expires_at 必须使用 App Store 返回的 expiresDate。
终身/非消耗型商品：expires_at 可以为空，表示长期有效。
人工赠送/补偿：expires_at 按后台填写的补偿截止时间判断。
```

如果用户订阅过期，老的 `sys_user_plan_snapshot` 不能继续参与权益保护。

如果用户过期后重新购买，后端必须：

```text
1. 使用本次新交易对应的 productId 解析最新 planCode。
2. 使用重新购买时后台最新 sys_plan_feature_rule 生成新的 feature_snapshot_json。
3. 将旧快照标记为 expired，或者保留 active 但因 expires_at < now 自动不参与计算。
4. 用户从新购买开始，按新快照和当前配置进行权益合并。
```

也就是说：

```text
老用户保护只保护“仍在有效期内的已付款承诺”。
过期后重新购买，视为新的购买合同，按新权益生效。
```

### 3.3 权益增加策略

如果新版本新增权益，例如：

```text
standard_single_child 新增 page_share
standard_single_child 新增 advanced_voice
standard_single_child 每日额度从 20 提升到 30
```

老用户应该自动获得新增或提升后的权益。

合并逻辑：

```text
enabled: old || current
limitValue: max(old, current)
accessLevel: 取更高权限
```

### 3.4 权益减少策略

如果新版本减少权益，例如：

```text
standard_single_child 历史周报从 4 周减少到 2 周
standard_single_child daily_capture 从 20 减少到 10
standard_single_child 关闭 advanced_voice
```

已付款老用户不能被减少。

合并逻辑：

```text
enabled: old 为 true 时继续 true
limitValue: old 大于 current 时保留 old
accessLevel: old 比 current 高时保留 old
```

### 3.5 免费用户策略

免费用户没有付款承诺，默认使用当前后台配置即可。

如果需要对某个免费用户单独补偿，则使用：

```text
sys_user_entitlement_grant
sys_user_feature_override
```

## 15. 用户权益决策合并算法

### 15.1 输入

```text
appCode
userId
当前时间 now
```

### 15.2 步骤

```text
1. 查询 sys_billing.getEntitlements(appCode, userId)，得到 active entitlementCode。
2. 根据 active entitlementCode 匹配当前 sys_membership_plan。
3. 如果没有付费 entitlement，则使用 free plan。
4. 查询当前 plan 的 sys_plan_feature_rule，得到当前配置权益。
5. 查询 sys_user_plan_snapshot 中用户 active 且未过期的购买时快照。
   - status = active。
   - starts_at <= now。
   - expires_at is null 或 expires_at > now。
   - 过期快照不得参与合并。
6. 将有效购买时快照和当前配置做“增益合并”：
   - 开关取更有利用户的值。
   - 额度取更大值。
   - 权限等级取更高值。
   - 如果没有有效购买快照，则只按当前配置和用户级赠送/覆盖计算。
7. 查询 sys_user_entitlement_grant 中 active 且未过期的赠送/补偿权益。
8. 将赠送/补偿权益对应的 plan 规则合并进结果。
9. 查询 sys_user_feature_override 中 active 且未过期的功能覆盖。
10. 应用用户级覆盖。
11. 输出 UserEntitlementDecisionView。
```

### 15.3 伪代码

```java
public UserEntitlementDecisionView resolveUserEntitlement(String appCode, Long userId) {
    EntitlementOverviewView billing = sysBillingService.getEntitlements(appCode, userId);
    List<String> activeCodes = activeEntitlementCodes(billing);

    MembershipPlan currentPlan = resolveCurrentPlan(appCode, activeCodes).orElse(freePlan(appCode));
    Map<String, FeatureAccess> currentRules = loadPlanRules(appCode, currentPlan.planCode());

    List<UserPlanSnapshot> snapshots = loadActiveSnapshots(appCode, userId);
    Map<String, FeatureAccess> protectedRules = mergeSnapshotsWithCurrent(currentRules, snapshots);

    List<UserEntitlementGrant> grants = loadActiveGrants(appCode, userId);
    Map<String, FeatureAccess> grantedRules = mergeGrants(appCode, protectedRules, grants);

    List<UserFeatureOverride> overrides = loadActiveOverrides(appCode, userId);
    Map<String, FeatureAccess> finalRules = applyOverrides(grantedRules, overrides);

    return toDecision(appCode, userId, currentPlan, activeCodes, finalRules);
}
```

## 16. 权益合并细则

### 16.1 enabled 合并

默认保护老用户：

```text
final.enabled = snapshot.enabled || current.enabled
```

如果用户级覆盖明确设置 enabled，则：

```text
final.enabled = override.enabled
```

但建议后台限制：对已付款权益做关闭操作时必须填写原因，并记录审计。

### 16.2 accessLevel 合并

等级顺序：

```text
none = 0
hidden = 1
preview = 2
limited = 3
full = 4
```

合并：

```text
final.accessLevel = max(snapshot.accessLevel, current.accessLevel)
```

用户覆盖：

```text
如果 override.accessLevel 非空，则使用 override.accessLevel
```

### 16.3 limitValue 合并

额度型权益默认取较大值：

```text
final.limitValue = max(snapshot.limitValue, current.limitValue)
```

用户覆盖：

```text
如果 override.limitValue 非空，则使用 override.limitValue
```

### 16.4 scope 合并

范围权限建议排序：

```text
account < single_child < per_child < family
```

合并：

```text
final.scopeCode = maxScope(snapshot.scopeCode, current.scopeCode)
```

### 16.5 过期处理

所有用户级记录都必须判断：

```text
status = active
starts_at <= now
expires_at is null or expires_at > now
```

## 17. 购买时权益快照生成时机

必须在以下时机生成或刷新：

1. 用户购买验证成功。
2. 用户恢复购买验证成功。
3. App Store Server Notification 确认订阅续费成功。
4. 手动刷新权益时发现 verified active 交易。
5. 客服人工补偿付费 plan 时。
6. 用户过期后重新购买成功时。

接入点：

```text
SysBillingService.projectEntitlementSnapshot
SysBillingService.refreshEntitlements
SysAppStoreNotificationService.ingest 后续投影逻辑
SysEntitlementCenterService.grantUserEntitlement
```

核心要求：

```text
只要产生 active paid entitlement，就生成 sys_user_plan_snapshot。
订阅商品的 sys_user_plan_snapshot.expires_at 必须等于 App Store claims.expiresDate。
终身/非消耗型商品可以 expires_at = null。
如果用户过期后重新购买，不允许复用旧 feature_snapshot_json，必须按重新购买时的最新权益规则生成新快照。
```

## 30. 最终落地结果

完成后，系统会具备：

```text
统一会员等级管理
统一权益功能管理
统一等级权益矩阵
统一商品映射
购买时权益快照
老用户权益保护
新增权益自动给老用户
人工赠送/补偿权益
用户级功能覆盖
后台修改审计
多 App 通用
前端 App 基本无感
```

最终判断链路：

```text
App 请求现有权益接口
  -> 后端读取 App Store 权益快照
  -> 后端读取当前会员等级配置
  -> 后端读取购买时权益快照
  -> 后端读取用户级赠送/补偿
  -> 后端读取用户级功能覆盖
  -> 后端合并出最终权益
  -> 后端转换为原 App 已有响应结构
  -> 前端无须大改即可使用
```

这就是推荐实施的统一后台式权益中心“模式三”。
