# paipaiV2 上线前完整匹配评估与 P0/P1 执行顺序（2026-04-24）

> 评估目标：按“第一个 iOS App 上线 Apple Store”标准，结合 `paipaiV2` 前端页面、SwiftUI/iOS 实现、新增需求、统一后端、数据库迁移、PowerSync 多 App 同步方案，判断拍拍伴读当前是否已经达到上线要求，并给出前后端数据库收口缺口清单、待办清单与 P0/P1 执行顺序。

---

## 0. 一句话结论

**当前不能签字认为“完全满足 Apple Store 上线合规要求”。**

更准确的状态是：

- **前端 × 后端 × 数据库的主干架构方向是正确的**：`appCode=paipai_readingcompanion`、统一后端、多 App app-definition、PowerSync 同步表、Apple 登录/订阅/账号删除/法务入口/OCR/TTS 服务端代理等关键模块都已经有实现。
- **代码级静态匹配度较高，但不是 100%**：核心接口和同步表基本对齐；但 iOS 侧存在疑似编译阻断、Release 占位符、本地 HTTP、语言/翻译/离线链路/隐私材料等上线级缺口。
- **“新增需求”未完全落地**：展示语言自由切换、iOS 17.4 翻译承诺、OCR 后分句与高亮、30 天使用时长留存、离线主链路、真实加密/隐私表述一致性仍需补齐。
- **后端不能证明当前环境可重新构建**：当前审计环境 `java` / `mvn` 不可用，只能看到历史 `target/surefire-reports` 中已有 41 个测试报告且未发现失败；这不能替代一次干净 CI 构建。

**上线判定：当前为“接近可联调/可收口”，不是“可直接提审”。**

## P0-4：展示语言“自由切换且准确”未完整达标

### 现象

当前 UI 文案实际主要是中/英二选一：

```swift
func uiText(_ zhHans: String, _ english: String) -> String
```

并且语言配置存在多个不统一来源：

- `UserPreference.default` 默认 `sourceLanguageCode = "en"`, `targetLanguageCode = "zh-Hans"`
- `languagePair(for:)` 中 `zh_to_en` 返回 `zh-Hans -> en`
- OCR 固定 `recognitionLanguages = ["zh-Hans", "en-US"]`
- TTS voice 只对 zh/en 特殊处理
- `AppBootstrap.default` supportedLocales 默认 `['zh-Hans','en']`

### 证据

- `Core/Models/PreferenceModels.swift`
- `App/PaipaiReadAlongV2App.swift`
- `Core/Services/OCRService.swift`
- `Features/Parent/LanguagePreferenceView.swift`

### 风险

- “多语言/18 种语言/自由切换”不能成立。
- 学习方向、OCR 语言、翻译源目标、TTS voice、UI locale 可能互相打架。

### 待办

- [ ] 建立统一语言配置模型：`uiLocale` / `sourceLanguageCode` / `targetLanguageCode` / `ocrLanguages` / `ttsVoice` / `translationAvailability` / `displayName`。
- [ ] 语言列表从后端 remote config/bootstrap 下发，前端只做 fallback。
- [ ] 修正默认学习方向：`zh_to_en` 与默认 source/target 必须一致。
- [ ] 每种支持语言补齐 UI 名称、OCR 支持、TTS voice、翻译可用性。
- [ ] App Store 文案只承诺真实可用的语言范围。

## P0-5：离线本地主要功能还没有完整闭环

### 现象

OCR/TTS 有设备端能力，但核心学习数据被登录态阻断：

- 创建孩子：未登录直接失败
- 保存句卡：未登录直接失败
- 使用时长：未登录不记录主链路数据
- PowerSync 获取凭证需要登录

### 证据

- `App/PaipaiReadAlongV2App.swift`
  - `createChildProfile` 有 `guard hasAuthenticatedSession`
  - `saveReviewCard` 有 `guard hasAuthenticatedSession`
  - `startUsageSession` / `endUsageSession` 有 `guard hasAuthenticatedSession`
- `Features/Capture/CaptureView.swift` 使用时长开始/结束也要求登录态
- `Core/Sync/PowerSyncConnector.swift` 登录态不足时不 fetch credentials

### 风险

- “后端离线本地 APP 主要功能可以正常使用”不成立。
- 后端不可用/用户未登录时，用户无法完整保存孩子与句卡。

### 待办

- [ ] 明确产品策略：是否允许“未登录离线本地模式”。
- [ ] 如果允许：本地创建 child/profile/card/usage，登录后 merge/claim/sync。
- [ ] 如果不允许：页面和 App Store 文案必须明确“首次使用需 Apple 登录，已登录后可离线复习/朗读”。
- [ ] 本地 only 数据与 PowerSync 同步表边界要明确，避免未登录数据误进服务端。

# 4. P1 缺口：建议提审前尽量清掉

## P1-1：OCR 后 NLTokenizer 分句未接入主流程

### 现象

`TextSegmentationService` 存在，但 OCR 结果页直接把整段 `recognizedText` 传给 `LearningDetailView`。

### 风险

- 多行/整页绘本文字会混在一起，影响翻译和朗读准确性。
- 与“拍一句/分句选择”的新增需求不完全一致。

### 待办

- [ ] OCR 后展示句子列表。
- [ ] 支持选择一句/多句。
- [ ] 保存选中句子的 source text / bounding boxes / language。

---

## P1-2：Vision 坐标高亮未落地

### 现象

OCR service 返回 `TextObservation.boundingBox`，但确认页只展示图片和 TextEditor，没有叠加坐标高亮。

### 待办

- [ ] 在图片预览上叠加 normalized bounding boxes。
- [ ] 点击 box 可选中文本片段。
- [ ] 高亮数据不进云端或仅存低敏必要字段。

---

## P1-3：使用时长 30 天留存策略没有主链路闭环

### 现象

已有旧/辅助 `UsageTrackingService` 的 30 天清理逻辑，但当前主链路是：

- iOS `UsageSessionRepository`
- PowerSync `reading_usage_session_v2`
- 后端 `ReadingUsageSessionV2Entity`

该主链路未看到本地和服务端统一 30 天自动删除/软删除策略。

### 待办

- [ ] iOS 启动时 purge 超过 30 天的本地 usage session。
- [ ] PowerSync 上传 tombstone 或后端定期删除/软删除。
- [ ] 后端加 DB retention job 或 SQL 运维脚本。
- [ ] 隐私政策写清保留期。

---

## P1-4：PowerSync user_preference 的本地 id 与服务端主键需要真联调

### 现象

Sync rules 对 `reading_user_preference` 下发：

```sql
SELECT 'me' AS id, ... FROM reading_user_preference
```

后端实体主键是 `userId`，iOS 本地表没有显式 `id` column 但 PowerSync 表隐含 id。这个设计可以成立，但必须通过真 PowerSync SDK 端到端测试证明：

- 下行 `id='me'` 能写入本地
- 本地修改 `me` 后上传 payload 能映射到服务端 `userId`
- 不会新增重复偏好记录

### 待办

- [ ] 做 user_preference 本地修改 → 上传 → 服务端更新 → 另一设备下行测试。
- [ ] 为该链路加自动化 contract test 或手工验收记录。

---

## P1-5：IAP / Paywall 展示与条款需复核

### 现象

前端默认套餐里有硬编码价格/产品 ID，Paywall 需要确保展示 StoreKit 返回的本地化价格、订阅周期、恢复购买、取消说明。当前条款里如写“订阅自动续费”，但产品如果是 lifetime，就会不一致。

### 待办

- [ ] StoreKit 产品类型、价格、周期与后端 plan catalog 对齐。
- [ ] Paywall 优先展示 `Product.displayPrice`。
- [ ] lifetime / monthly / yearly 的条款分开。
- [ ] 恢复购买入口明显可见。
- [ ] App Store Review Notes 写清购买/恢复/账号删除路径。

---

## P1-6：Entitlements 可能有冗余 capability

### 现象

iOS entitlements 中存在 iCloud/CloudKit、App Groups、IAP 等能力，但当前主同步方案是 PowerSync。未使用 capability 会增加审核和配置复杂度。

### 待办

- [ ] 只保留真实使用的 capability。
- [ ] 如果不使用 CloudKit，移除 iCloud/CloudKit。
- [ ] 如果使用 App Groups，说明用途；否则移除。

---

## P1-7：硬编码与模板化残留需要清理

### 现象

仍能看到多处硬编码/默认值：

- iOS `project.yml` 的本地 API Base URL
- `AppIdentity.appCode` / fallback bundle id
- 默认套餐价格和产品 ID
- OCR 语言固定 `zh-Hans/en-US`
- UI supportedLocales fallback 只有中英
- 后端 provider 类名和默认 DashScope endpoint/model
- sampleCards / unused service 等样例残留

### 判断

不是所有硬编码都必须消灭。**AppIdentity.appCode、fallback bundle id 可以作为 target 编译常量存在**；但 Release 配置、价格、语言、provider、法律 URL、公告、套餐文案应从后端配置或 StoreKit 获取。

### 待办

- [ ] 清单化“允许硬编码”和“禁止硬编码”。
- [ ] Release 禁止硬编码：Team、Bundle、版本、API URL。
- [ ] 商业配置禁止硬编码：价格、权益、商品展示。
- [ ] 语言/provider 从 remote config/bootstrap 下发。
- [ ] 删除未接入主链路的 sample/unused 代码，或明确标注 debug-only。

# 10. 最终建议

如果目标是“尽快可提审”，不要继续发散功能；建议按以下最短路径：

1. **先修编译 + 建 release preflight**。
2. **收窄产品承诺**：如果多语言、iOS 17.4 翻译、未登录离线保存来不及，就不要在首版承诺。
3. **补齐 Apple / App Store / HTTPS / 法务配置**。
4. **完成 Apple 登录、购买恢复、账号删除、OCR/TTS、PowerSync 的真实验收**。
5. **最后再做 P1 体验增强**：分句、高亮、多语言扩展、留存自动清理、模板化新 App。

**当前项目的架构方向是对的，但上线前不能只看“接口和表已经有了”。Apple Store 首发更需要：真编译、真配置、真法务一致性、真链路验收，以及不要承诺尚未完整实现的语言/离线/加密能力。**
